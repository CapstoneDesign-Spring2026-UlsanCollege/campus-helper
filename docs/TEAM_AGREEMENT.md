## Team Roles

Roles rotate every sprint so everyone experiences each responsibility.

| Role | Responsibilities |
| --- | --- |
| PM | Coordinate sprint tasks, keep GitHub work visible, submit sprint packet |
| Scribe | Record decisions, update documentation, track blockers |
| QA Lead | Verify working features, test demos, review acceptance criteria |
| Demo Driver | Prepare and run demos, maintain backup demo plan |
| Developer | Implement assigned issues and contribute reviewed code |

## Communication

| Tool | Purpose |
| --- | --- |
| KakaoTalk | Quick team communication |
| Discord | Meetings and discussion |
| GitHub Issues | Task management and evidence tracking |

The team agrees to respond within 24 hours, communicate blockers early, and notify teammates when a meeting cannot be attended.

## Meeting Expectations

Meetings happen Wednesday after class, plus online meetings when needed. Meetings should assign clear owners, focus on active issues, and end with visible GitHub progress.

## GitHub Workflow

```text
Issue -> Branch -> Pull Request -> Review -> Merge -> Evidence
```

- No major work without an issue.
- Use branches for feature work.
- Use pull requests for merges.
- Important work should be reviewed by a teammate.

## Evidence Rule

If it is not linked, it did not happen. Each student should post 2-3 receipts each week, such as commits, PRs, issues, review comments, screenshots, or documentation updates.

## Conflict Resolution

1. Discuss the issue respectfully within the team.
2. Adjust responsibilities if needed.
3. Document the issue in GitHub.
4. Ask the instructor for help if the problem continues.

## Decision Rules

- Most decisions: majority vote.
- Major project changes: full team agreement.
